# pipeline.config.json — Base Schema Specification

This document defines every field in `pipeline.config.json`. It is the
canonical reference for all builders that read config.

Generated by `/new-project` based on user selections. Never edited
manually after generation except through a guided update command.
Never contains secrets — all credentials live in `.env`.

---

## Extensibility Note

Valid values for `frontend`, `backend`, and `deploymentTarget` reflect
the stacks supported in v1. The schema itself is not the constraint —
the framework is designed to be extended.

Adding a new stack requires exactly three things:
1. Add the new value to the valid values list in this document
2. Add a matching security checklist file in `.claude/checklists/`
3. Add a component library mapping in the design system table

Nothing in the core pipeline changes. Builders read whatever value is
in config and adapt. A value not on the v1 list won't break the
pipeline — it just won't have a checklist or component library mapping
until one is added.

## Base Fields (Every Project)

These fields are present in every config regardless of stack, entry
path, or project type.

| Field | Type | Valid Values | Notes |
|---|---|---|---|
| `name` | string | Any | Project name. Lowercase with hyphens. e.g. `my-todo-app` |
| `type` | string | `mobile_app` `web_app` `marketing_site` `email` `sms` `full_stack` | Determines which stack extensions are included |
| `entryPath` | string | `new` `migration` | Set at start of /new-project. Never changes after generation |
| `repoURL` | string | Valid GitHub or GitLab URL | Full HTTPS URL. e.g. `https://github.com/user/repo` or `https://gitlab.com/user/repo` |
| `repoProvider` | string | `github` `gitlab` | Determines which CI/CD platform the Cloud Architect provisions. |
| `frontend` | string or null | `react-native-expo` `react-vite` `nextjs` `react-email` `null` | null for backend-only or SMS projects |
| `backend` | string or null | `supabase` `planetscale` `neon` `python-fastapi` `vercel-edge` `none` `null` | null if not yet determined |
| `emailProvider` | string or null | `resend` `sendgrid` `aws-ses` `null` | null if project does not use email |
| `smsProvider` | string or null | `twilio` `vonage` `aws-sns` `null` | null if project does not use SMS |
| `deploymentTarget` | string or null | `vercel` `expo-eas` `railway` `fly-io` `aws` `null` | null until confirmed during /setup-pipeline |
| `createdAt` | string | ISO 8601 datetime | Set once at generation. Never updated |
| `updatedAt` | string | ISO 8601 datetime | Updated on every config change |
| `setupComplete` | boolean | `true` or omitted | Set to `true` by Cloud Architect at end of `/setup-pipeline`. The VS Code panel reads this to determine whether setup has run. Omitted (not false) before setup completes. |

## Stack Extensions

Stack extensions are nested under a `stack` object. Fields vary by
project type and selections. Unused fields are null, never omitted.

### `stack.brandSystem`
Path to the brand design system file if one has been configured.
Set during `/setup-pipeline` brand setup step or after first
feature design if the user opts to capture their brand decisions.

| Valid values | `"brand.md"` `null` |
|---|---|

### `stack.componentLibrary`
Which component library was provisioned by the Cloud Architect.
Matches the stack table in the design spec.

| frontend value | componentLibrary |
|---|---|
| `react-vite` | `shadcn-ui` |
| `nextjs` | `shadcn-ui` |
| `react-native-expo` | `nativewind` |
| `react-email` | `react-email` |
| null | null |

### `stack.designTokens`
Where design tokens are configured for this project.

| frontend value | designTokens |
|---|---|
| `react-vite` | `src/globals.css` |
| `nextjs` | `src/globals.css` |
| `react-native-expo` | `theme.ts` |
| `react-email` | null |
| null | null |

### `stack.testRunner`
Test framework for the QA builder to use.

| stack | testRunner |
|---|---|
| `react-vite` | `vitest` |
| `nextjs` | `jest` |
| `react-native-expo` | `jest` + `detox` |
| `python-fastapi` | `pytest` |
| `supabase` (edge functions) | `deno test` |

### `stack.cicd`
CI/CD platform configured during /setup-pipeline. Determined by
`repoProvider` — GitHub repos use GitHub Actions, GitLab repos use
GitLab CI.

| Valid values | `github-actions` `gitlab-ci` `null` |
|---|---|

### `stack.environments`
Named deployment environments. Set during /setup-pipeline.

```json
"environments": {
  "dev": "branch name or URL",
  "staging": "URL or null",
  "prod": "URL or null"
}
```

## Entry Path Differences

The two entry paths produce configs with the same shape but different
field populations at generation time.

| Field | `new` path | `migration` path |
|---|---|---|
| `name` | User-defined | Taken from existing project |
| `entryPath` | `"new"` | `"migration"` |
| `repoURL` | New repo URL | Existing repo URL |
| `frontend` | Selected during Q&A | Detected from existing codebase |
| `backend` | Selected during Q&A | Detected from existing codebase |
| `stack.componentLibrary` | Set during /setup-pipeline | Detected or set during /setup-pipeline |
| `stack.environments` | null until /setup-pipeline | May be pre-populated from existing setup |

---

## Rules for Builders

1. **Never assume a field value.** Always read from config.
2. **Null means not applicable or not yet set.** Do not treat null as an error.
3. **Never write to pipeline.config.json.** Only `/new-project` and `/setup-pipeline` write this file.
4. **Always check `entryPath`** when behavior differs between new and migration projects.
5. **Stack extensions live under `stack`.** Never reference stack fields at the root level.
